# ADR-007 Testcontainers와 k6 테스트 전략

## 상태와 결정일

Accepted. 결정일: 2026-07-19.

## 맥락과 문제

`SELECT ... FOR UPDATE` 같은 락 동작은 DB 벤더마다 미묘하게 다르게 동작할 수 있어, 동시성·정합성 검증의 신뢰도가 테스트 DB 선택에 좌우된다. 또한 "다수 인스턴스에서도 문제 없어야 한다"는 요구사항은 단일 프로세스 테스트만으로는 증명되지 않는다.

## 결정 동인

- 락·트랜잭션 검증은 실제 운영 DB(MySQL)와 동일한 동작을 보장해야 한다.
- 동시성 테스트는 재현 가능한 수치(초기 잔액, 동시 요청 수, 기대 성공/실패 건수)로 고정해야 한다.
- 부하 상황에서의 실제 동작(응답시간, 성공/실패 분포)을 관찰할 수 있어야 한다.

## 검토한 선택지

| 선택지 | 신뢰도 | 실행 속도 | 판단 |
| --- | --- | --- | --- |
| H2 인메모리 DB | 락 동작이 MySQL과 다를 수 있어 신뢰도가 낮다. | 빠르다. | 제외. 동시성 검증 목적에는 부적합하다. |
| Testcontainers(MySQL) | 실제 MySQL과 동일한 락 동작을 검증할 수 있다. | H2보다 느리지만 감내할 수준이다. | 채택. |
| 부하 검증 없이 단위/통합 테스트만 | 코드 레벨 정확성은 검증되지만 실제 부하 동작은 알 수 없다. | 가장 빠르다. | 보류. 도전 요구사항(동시성) 근거로는 부족하다. |
| k6 부하 테스트 추가 | 동시 요청 상황에서의 실제 처리량·응답시간·성공/실패 분포를 관찰할 수 있다. | 별도 실행 단계가 필요하다. | 채택(보완). |

## 결정과 이유

동시성·락 통합 테스트는 Testcontainers로 띄운 실제 MySQL을 사용한다. 핵심 시나리오는 수치로 고정한다: 초기 포인트 10,000P, 메뉴 가격 4,000P, 동시 주문 요청 10개(같은 userId) → 기대 성공 2건, 기대 실패 8건, 최종 잔액 2,000P, 음수 잔액 없음(`docs/order-policy.md`). 여기에 더해 k6로 동시 요청 부하 시나리오를 만들어, 다수 인스턴스 환경(docker-compose로 앱 컨테이너 2개 이상)에서도 같은 결과가 유지되는지 별도로 검증한다.

## 결과와 단점

락 검증의 신뢰도가 높아지고, 도전 요구사항(동시성)에 대한 정량적 근거를 남길 수 있다. 반면 Testcontainers는 H2보다 테스트 실행이 느리고, k6 다중 인스턴스 검증은 CI에 상시 포함하기보다 필요할 때 수동으로 재현하는 별도 절차로 관리해야 한다.

## 검증 현황과 계획

- 실제 근거: 없음 (스켈레톤 단계, 테스트 코드 미작성).
- 계획된 검증: `docs/order-policy.md`의 동시성 시나리오를 Testcontainers 통합 테스트로 구현하고, k6 스크립트로 단일/다중 인스턴스 부하 결과를 비교 기록한다.

## 재검토 조건

- Testcontainers 실행 시간이 개발 루프를 과도하게 늦출 때, 일부 검증을 focused test로 축소할 근거가 생길 때.
