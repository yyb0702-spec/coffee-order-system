# ADR-006 Idempotency Key 도입

## 상태와 결정일

Accepted. 결정일: 2026-07-19.

## 맥락과 문제

네트워크 재시도(클라이언트 타임아웃 후 재요청, 다수 인스턴스 뒤 로드밸런서의 재시도 등)로 같은 주문/충전 요청이 두 번 전송될 수 있다. 동시성 락(ADR-002)은 "같은 시점의 경합"은 막지만 "시간차를 둔 중복 요청"은 막지 못한다.

## 결정 동인

- 다수 인스턴스 환경에서도 중복 요청을 안전하게 차단해야 한다.
- 조회 후 삽입(TOCTOU) 방식의 레이스 컨디션을 피해야 한다.

## 검토한 선택지

| 선택지 | 다수 인스턴스 안전성 | 구현 복잡도 | 판단 |
| --- | --- | --- | --- |
| 미도입 (동시성 제어만으로 방어) | 시간차를 둔 재시도는 막지 못한다. | 추가 구현이 없다. | 제외. 재시도로 인한 이중 처리 위험이 남는다. |
| 애플리케이션에서 "조회 후 없으면 insert" | 조회와 insert 사이 레이스가 생겨 두 인스턴스가 동시에 통과할 수 있다. | 구현은 단순하지만 정확하지 않다. | 제외. TOCTOU 레이스. |
| unique 제약 테이블에 즉시 insert 시도, 실패 시 중복 처리 | DB unique 제약이 원자성을 보장해 다수 인스턴스에서도 안전하다. | 제약 위반 예외를 잡아 에러 코드로 변환하는 처리가 필요하다. | 채택. |

## 결정과 이유

`idempotency_key`(`idempotency_key` unique, `endpoint`, `created_at`) 테이블을 두고, 클라이언트가 `Idempotency-Key` 헤더로 전달한 값을 본 비즈니스 로직과 같은 트랜잭션 안에서 즉시 insert 시도한다. 삽입이 성공하면 로직을 진행하고, unique 제약 위반이면 `DUPLICATE_REQUEST`(409)로 응답한다. 같은 트랜잭션에 두는 이유는, 별도 트랜잭션으로 먼저 커밋해버리면 본 로직이 실패해 롤백됐을 때 idempotency key만 남아 해당 요청이 영구히 재시도 불가능해지기 때문이다.

## 결과와 단점

다수 인스턴스에서도 조회-삽입 사이의 레이스 없이 중복 요청을 차단할 수 있다. 반면 모든 상태 변경 API가 `Idempotency-Key` 헤더를 필수로 요구하게 되어 클라이언트 쪽에서 키 생성·관리 책임이 생기고, 오래된 키를 정리하는 정책(TTL 등)이 별도로 필요하다.

## 검증 현황과 계획

- 실제 근거: 없음 (스켈레톤 단계. 컨트롤러는 헤더만 받고 있고 실제 중복 검사 로직은 TODO로 남아 있다).
- 계획된 검증: 같은 `Idempotency-Key`로 동시에 두 번 요청했을 때 하나만 처리되고 나머지는 `DUPLICATE_REQUEST`를 받는지 통합 테스트로 확인한다.

## 재검토 조건

- 오래된 idempotency key의 정리(TTL/배치 삭제) 정책이 필요해질 때.
- 헤더 강제가 클라이언트 구현 부담을 과도하게 높인다는 판단이 설 때.
