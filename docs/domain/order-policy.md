# 주문 정책

## 기본 규칙

- 주문/결제는 포인트로만 처리한다.
- 주문 생성과 포인트 차감(원장 insert 포함)은 하나의 DB 트랜잭션으로 처리한다.
- 주문에는 결제 금액 스냅샷 `paidAmount`를 저장한다 (메뉴 가격이 나중에 바뀌어도 주문 당시 금액 보존).
- 주문 상태(`status`)는 `PAID`, `CANCELLED` 등 다중 값으로 설계한다. 검증 실패 시 결제 완료 주문을 생성하지 않는다.

## 동시성 제어

두 장치를 역할 분리하여 병행한다.

1. **Redisson 분산 락**: `userId` 기준으로 주문 요청 진입 자체를 직렬화한다. 동일 유저의 동시 요청이 여러 개 들어와도 하나씩만 뒤 로직(DB 트랜잭션)에 진입하도록 앞단에서 걸러, 불필요한 DB 락 대기/트래픽을 줄인다.
2. **DB 비관적 락(`SELECT FOR UPDATE`)**: `user_point` row를 잠그고 잔액 확인 → 차감 → 원장 insert → 주문 insert까지를 하나의 임계구역으로 묶어 최종 정합성을 보장한다.

정확성의 최종 책임은 DB 락에 있고, Redisson은 트래픽/중복요청 방어용 필터라는 점을 구현 시 유지한다.

Redisson 락 획득에 실패(타임아웃)하면 DB 트랜잭션까지 진입하지 않고 즉시 `ORDER_LOCK_NOT_ACQUIRED`(409)로 응답한다. 클라이언트는 이 응답을 받으면 재시도 여부를 판단한다.

### 동시성 테스트 시나리오 (예시)

구현 검증 시 아래처럼 수치를 고정해 재현 가능한 테스트로 만든다.

```text
초기 포인트: 10,000P
메뉴 가격: 4,000P
동시 주문 요청: 10개 (같은 userId)
기대 성공: 2개
기대 실패: 8개 (INSUFFICIENT_POINT 또는 ORDER_LOCK_NOT_ACQUIRED)
최종 잔액: 2,000P
음수 잔액: 절대 발생하지 않아야 함
```

## 중복요청 방지

- 클라이언트가 요청마다 고유 Idempotency Key(예: UUID)를 헤더로 전달한다.
- 서버는 해당 키를 unique 제약이 걸린 저장소로 기록해 중복 요청을 차단한다.
- 네트워크 재시도로 인한 이중 주문/이중 결제를 막는 것이 목적이며, 동시성 제어(락)와는 별개의 방어선이다.

- 주문/결제가 트랜잭션 커밋된 후 `OrderCompletedEvent`를 Kafka로 발행한다.
- Topic: `order.completed`.
